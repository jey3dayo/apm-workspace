# Simple Mode 実行ガイド

迅速で実用的な問題発見に特化したシンプルモードの実行方法を定義します。

## 概要

Simple Mode は、短時間で重大な問題を拾うためのレビューです。理想は 4 観点の並列確認ですが、並列手段が無ければ同じ観点を順番にレビューして構いません。

対象観点:

- セキュリティ
- パフォーマンス
- コード品質
- アーキテクチャ

## 実行フロー

### Step 1: 対象の確定

- 明示フラグがあればそれを優先する: `--staged` / `--recent` / `--branch <name>`
- フラグが無ければ次の順に対象を探す:
  - staged changes
  - previous commit diff
  - primary development branch との差分
  - 最近変更された reviewable files
- primary development branch は、明示指定が無ければ remote の default branch、次に `main`、最後に `master` の順で解決する
- 生成物・ロックファイル・vendor・build artifact は対象から外す
- reviewable files は、人が保守する source / config / test / 関連度の高い docs を指す
- レビューのために `git add` や `git commit` は行わない

### Step 2: 4 観点の確認

環境が許せば 4 観点を並列で確認し、無理なら順番に確認する。どちらの場合も出力要件は同じ。

各観点の最低チェック:

- セキュリティ:
  - 入力検証
  - 認証・認可
  - 機密情報の扱い
  - 既知の脆弱性パターン
- パフォーマンス:
  - 明らかな N+1 / 非効率ループ
  - 不要な再計算
  - メモリ保持やリークの懸念
- コード品質:
  - 過度な複雑度
  - 重複コード
  - 命名不整合
  - マジックナンバーや不明瞭な分岐
- アーキテクチャ:
  - 責務混在
  - 依存方向の不整合
  - 循環依存
  - 拡張しにくい構造

### Step 3: 集約ルール

- 同じ問題を複数観点で検出した場合は 1 件に統合する
- severity が衝突した場合は高い方を採用する
- セキュリティ指摘は exploitability があるなら 1 段階引き上げてよい
- すべての指摘に `file:line` を付ける
- 根拠が弱い推測は断定せず、「懸念」「確認事項」として表現する

### Step 4: 出力形式

simple mode は findings first で返す。

- simple mode では severity を主軸にし、⭐️評価は出さない

```markdown
## 発見事項

### 高優先度

- [カテゴリ] 問題の要約 — `path/to/file.ts:42`
  - 影響
  - 修正方針

### 中優先度

- ...

### 低優先度

- ...

## 即座に直せる項目

- すぐに修正できる内容だけを列挙

## 推奨アクション

- 次に取るべき行動を優先順に列挙
```

findings が無い場合は「重大な指摘なし」と明示し、残る確認リスクがあれば短く添える。
