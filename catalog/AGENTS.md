# AI 開発協働ガイド

## Definition of Done (DoD)

すべてのタスク完了時に、作業内容に該当する以下の確認を満たすこと。リポジトリで定義された check / test / format タスクがある場合はそれを優先する。

ただし、通常作業のたびに full gate を機械的に実行しない。トークン・時間・環境揺れを抑えるため、作業中は変更範囲に応じた軽い確認を優先し、push / PR / deploy など外部共有やロールアウトの直前に full gate を実行する。

### 1. 品質チェック（該当時必須）

- 型チェック対象の変更: 型エラー 0 件
- リント対象の変更: リント違反 0 件
- テスト対象の変更: 関連テスト成功
- フォーマット対象の変更: フォーマッター適用済み、または format check 成功
- push / PR 作成前: リポジトリ定義の format タスクで auto-format を実行する。auto-format が存在しない場合のみ format check 成功で代替する

### 2. 軽い確認と full gate の使い分け

| タイミング                                           | 実行する確認                 | 例                                                               |
| ---------------------------------------------------- | ---------------------------- | ---------------------------------------------------------------- |
| 実装中・小さな修正後                                 | 変更範囲に直結する軽い確認   | touched file の formatter、関連テスト、`git diff --check`        |
| 型・lint・テストに影響する変更後                     | 該当領域の focused check     | 対象 package の typecheck / lint / unit test                     |
| 共通基盤・依存・設定・生成物・永続データ構造の変更後 | 早めに full gate へ昇格      | repo 定義の `check` / `test` / `ci` / `verify`                   |
| commit 前                                            | 差分が壊れていないことの確認 | `git diff --check`、staged/touched files の format / lint        |
| push / PR / deploy 前                                | full gate                    | repo 定義の auto-format、format check、lint、test、build、verify |

full gate を実行しない場合は、報告時に実行した軽い確認と、full gate を省略した理由を短く明示する。

### 3. スコープ遵守

- 指示された範囲のみ変更する
- 余計な機能追加・リファクタリング・ドキュメント生成を行わない
- 既存ファイルの編集を優先し、新規ファイル作成は必要最小限

### 4. 再現可能性

- 変更後のビルド・テストが第三者環境でも成功すること
- 環境依存の暗黙知を残さない

## 停止・確認ポリシー

ユーザーが「実装して」「直して」「進めて」など実作業を明示した場合は、原則として追加承認を待たずに実装、該当する品質確認、報告まで進める。

承認確認や作業停止は、以下のいずれかに該当する場合に限る。

- 破壊的操作、データ削除、履歴改変、外部公開、費用発生を伴う
- 仕様判断が複数あり、選択によってユーザー体験や互換性が大きく変わる
- 大規模な設計変更、依存追加、永続データ構造変更、セキュリティ境界変更を伴う
- 既存のユーザー変更と衝突し、どちらを優先すべきか判断できない
- 必要な権限、認証、秘密情報、外部サービス状態が不足している
- 同一アプローチで **3回失敗** した

`main` / `master` ブランチ上では、`git restore`、`git checkout --`、`git reset --hard` など作業ツリーの変更を破棄・復元する操作を行わない。必要に見える場合でも、まず `git status --short` で対象を確認し、ユーザーに明示確認する。

同一アプローチで3回失敗した場合は、それ以上リトライせず以下を報告する。

1. 試行内容: 何を3回試みたか
2. 失敗理由: 各回の具体的エラー
3. 代替案: 別のアプローチの提案（あれば）

## 実装ワークフロー

1. 計画 - 変更概要を短く共有し、停止・確認ポリシーに該当する場合のみ承認を得る
2. コード理解 - ツール優先順位に従い効率的に探索
3. 実装 - 型安全・影響範囲確認
4. 品質保証 - DoD の該当項目を実行
5. 独立レビュー - 変更を通常のコードレビュー観点で確認する。手動差分確認を基本とする。指摘があれば修正を適用し、最大3回繰り返す
6. 報告 - 完了報告、問題があれば早期相談

## 重要な開発原則

- 型安全性: any 型・型アサーション排除
- エラーハンドリング: 適切なエラー処理パターン
- 未使用変数: 原則削除。例外的に `_` プレフィックス許可（catch 文、フレームワーク必須引数のみ）

## モデル運用方針（Claude × Fable）

Claude Code を Fable モデルで動かす場合、役割を以下のように分担する。

- 構想・指揮系統は Fable が担う
  - 要件整理、設計方針の決定、タスク分解、オーケストレーション、レビュー判断
- 実装は Codex MCP / Opus / Sonnet に委譲する
  - コード生成・編集などの実作業は、Fable から該当モデルへ委任して実行する
- Fable は全体の意思決定と進行管理に専念し、実装の細部は委譲先の出力を検証・統合する役割に徹する

## コード探索の原則

**優先順位**（トークン効率重視）:

1. `rg` - パターン検索・横断的調査
2. `sed` / `nl` / 専用 Read ツール - 設定ファイル・小規模ファイルの確認

### CLI が見つからない場合

- まず `command -v <tool>` または `which <tool>` で PATH 上の存在を確認する
- リポジトリの `mise.toml`、`mise.lock`、task 定義でその tool が mise 管理か確認する
- mise 管理なら `mise which <tool>` で解決先を確認し、未導入なら `mise install` を検討する
- それでも使えない場合のみ、理由を報告して raw command や代替コマンドに fallback する

## ファイル操作原則

- 既存ファイル編集を優先、新規作成は必要最小限
- README.md や \*.md は明示的に要求された場合のみ作成
- 既存テストなど重要なものは勝手に削除しない
- ユーザー未確認の変更を `git restore` などで復元・破棄しない。特に `main` / `master` 上では restore 系操作を実行しない
- スキル実行や調査で作る一時的な出力、調査計画、調査メモ、レビュー結果、レポート、スクリーンショット、作業用生成物は、ユーザーが保存先を指定していない限りリポジトリ直下の `tmp/` 配下に集約する
- `tmp/` 配下の出力は原則コミットしない。成果物として永続化が必要なものだけ、目的に合う正規のパスへ移してからコミット対象にする
- スキルの手順が `research_*` や `reports/` などの相対出力先を要求する場合も、まず `tmp/<skill-or-topic>/` 配下へ読み替える。ただしユーザーが明示した保存先、既存ツールが要求する固定パス、プロダクトに組み込む実成果物はこの限りではない

## ドキュメント作成の優先順位

新しい知識を追加するとき:

1. Skill - 繰り返し使う知識
2. Agent - 自動実行すべきタスク
3. Command - ユーザーが手動実行する操作
4. Rules/Steering - プロジェクト固有のルール
5. llms.txt - agent 向けの短い入口・索引
6. Docs - 上記で表現できない場合のみ、最小限

## MCP 配置方針

Codex の global MCP は、リポジトリをまたいで常時使う基盤だけに限定する。

- global に残すもの:
  - `mcp-simple-voicevox` - 通知
  - `codex` - Codex 連携
  - `jina-reader` - 調査
  - `context7` - ライブラリ・SDK・CLI の current docs 確認
- global に置かないもの:
  - `tauri-mcp-server` - Tauri repo 限定
  - `peekaboo` - 画面確認・スクリーン操作が必要なときだけ
- ブラウザ操作のデフォルト:
  - `claude-in-chrome`（Chrome 拡張連携）または Codex の Chrome アドオン - クリック・フォーム入力・スクショ・コンソール確認など通常操作
  - `chrome-devtools`（CDP 経由）- Lighthouse 監査・パフォーマンストレース・ヒープスナップショットなど DevTools 固有の深い分析が必要な場合のみ repo-local で追加
- 調査要求は `cross-research` の考え方で source type を先に分類する
  - ライブラリ・SDK・CLI・クラウドサービスの current docs 確認は `context7` を優先する
  - 一般 Web 検索、URL 読み取り、公式ページ・記事・ブログ取得は `jina-reader` を使う
  - 複数サブトピックに分かれる広い調査や比較は `web-research` を使う
  - GitHub、social、video、feed、authenticated source など source-specific な到達性が必要な場合は専用 connector / Agent Reach 系を検討する
- MCP 配置を変えるときは、APM などの source of truth を編集して再生成する
  - `~/.codex/config.toml` のような deployed target を直接編集しない
- repo-local MCP 候補をここに固定リストとして持たない（陳腐化するため）
  - リポジトリ一覧は `ghq list -p` を SSoT として使う
  - 個々の repo-local MCP 実体は各リポジトリの `apm.yml` を確認する

## Git コミット規約

- 変更内容を簡潔に記述するのみ
- 追加の署名やフッターは不要

## Git Worktree 方針

- 実装開始時に隔離 workspace が必要かを判断する workflow は `using-git-worktrees` スキルが担当する
- `git wt` / `git worktree` の操作仕様、`.worktrees/` 運用、作成・削除・診断手順は `git-worktree` スキルを source of truth とする
- コミット分割とコミットメッセージ作成は `atomic-commit` スキルが担当し、worktree 作成・切替・削除の判断を持たせない
- AGENTS.md や個別スキルには詳細な `git wt` 手順を重複させず、必要な場合は `git-worktree` スキルへの参照にとどめる

## コミュニケーション原則

- 分からないことは素直に報告し、思考プロセスをオープンにする
- 内部で英語思考してもよいが、最終回答は必ず日本語
- 不確実性・問題発見時は早期相談
- 番号だけの回答を求める場合は、同じメッセージ内に `1.` `2.` `3.` の各選択肢本文を必ず再掲する

### 確認時の返答

- 確認や小さな判断を求めるときは、短く答えられる聞き方を優先する
- ユーザーは `y/n`、番号、短い語句だけで返答してよい
- 短い確認で足りる場面では、長い説明を求めない

### Voice Notification Rules

設定:

- 音声通知の設定は `speaker=1` `speedScale=1.3` `async=true` を使う
- すべてのタスク完了時には必ず `mcp-simple-voicevox` で音声通知を行う
- 重要なお知らせやエラー発生時にも `mcp-simple-voicevox` で音声通知を行う
- `mcp-simple-voicevox` tool が露出していないターンでは、音声通知を省略し、未実行だったことを報告しない

文面:

- 英単語は適切にカタカナへ変換してから `VOICEVOX` に送る
- `VOICEVOX` に送るテキストは不要なスペースを削除する
- 1回の音声通知は 100 文字以内で、結果だけをシンプルに伝える
- 詳しい技術的説明は音声通知に含めず、結果のみを簡潔に報告する

タイミング:

- 音声通知のタイミングは、命令受領時、作業開始時、作業中、進捗報告時、完了時を基本とする

例:

- 命令受領時の例は「了解です」「承知しました」
- 作業開始時の例は「〜を開始します」
- 作業中の例は「調査中です」「修正中です」
- 進捗報告時の例は「半分完了です」「もう少しです」
- 完了時の例は「完了です」「修正完了です」

## 禁止事項

- `find`, `grep`, `cat`, `head`, `tail` の直接使用は原則避け、`rg`、専用ツール、`sed` / `nl` を優先する
- 要求されていない機能追加・リファクタリング
- 既存テスト・重要ファイルの無断削除
- any 型の導入

## スキル実行時のパス解決

外部スキル（`openai/skills`, `vercel-labs/*`, その他公開スキル）が相対パスでファイルを参照する場合:

- 基準ディレクトリ:
  - Claude Code: `~/.claude/skills/<skill-name>/`
  - Codex: `~/.agents/skills/<skill-name>/`
- `~/.codex/skills` は legacy であり、Codex の active deployment root として扱わない
- 解決方法: 相対パス `scripts/foo.py` → 絶対パス `<基準ディレクトリ>/scripts/foo.py`

#### 実行例 (gh-address-comments)

```bash
# Claude Code
python3 ~/.claude/skills/gh-address-comments/scripts/fetch_comments.py

# Codex
python3 ~/.agents/skills/gh-address-comments/scripts/fetch_comments.py
```
